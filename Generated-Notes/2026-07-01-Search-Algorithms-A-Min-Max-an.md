## 1. 🧱 The Core Concept (Basics Refresh)

### The Search Taxonomy

At the scale of FAANG infrastructure, search algorithms are not academic exercises; they are core engines driving logistics, game theory, network routing, and compiler optimization. We categorize these searches based on **information availability**, **adversarial dynamics**, and **memory limits**.

```
                           [Search Algorithms]
                                    |
         +--------------------------+--------------------------+
         |                                                     |
  [Single-Agent Pathfinding]                            [Adversarial Games]
         |                                                     |
   +-----+-----+                                         +-----+-----+
   |           |                                         |           |
[Uninformed] [Heuristic]                              [Deterministic] [Stochastic]
 (BFS/DFS)     (A*, IDA*, Beam)                        (Minimax, α-β)  (Expectiminimax)
```

### Core Search Algorithms Comparison

| Algorithm | Time Complexity (Worst) | Space Complexity (Worst) | Optimal? | Complete? | Real-World Archetype |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **A\*** (Consistent $h$) | $O(b^d)$ | $O(b^d)$ | **Yes** | **Yes** | Uber/Ride-share routing engines |
| **A\*** (Inadmissible $h$) | $O(b^d)$ | $O(b^d)$ | No | **Yes** (on finite graphs) | Suboptimal high-speed routing |
| **Minimax** | $O(b^d)$ | $O(d)$ | **Yes** (against optimal opponent) | **Yes** (finite tree) | Classic chess/checkers engines |
| **Alpha-Beta Pruning** | $O(b^d)$ | $O(d)$ | **Yes** | **Yes** | Production game AI engines |
| **Alpha-Beta (Best Order)** | $O(b^{d/2})$ | $O(d)$ | **Yes** | **Yes** | Optimized game search with move ordering |
| **Beam Search** | $O(B \cdot d)$ | $O(B)$ | No | No | LLM Token Generation, Machine Translation |
| **IDA\*** | $O(b^d)$ | $O(d)$ | **Yes** (if $h$ is admissible) | **Yes** | Embedded/Memory-constrained robotics |

> **Key:** $b$ = branching factor, $d$ = search depth, $B$ = beam width.

### Mathematical Guarantees

#### 1. Admissibility
A heuristic $h(n)$ is **admissible** if it never overestimates the actual cost to reach the goal state $g$. 
$$\forall n, \quad 0 \leq h(n) \leq h^*(n)$$
where $h^*(n)$ is the true shortest path cost from node $n$ to the goal. 

* **Consequence:** In tree-search configurations, admissibility guarantees that A\* will find the optimal path first.

#### 2. Consistency (Monotonicity)
A heuristic $h(n)$ is **consistent** (or monotonic) if, for every node $n$ and every successor $n'$ generated by action $a$ with cost $c(n, a, n')$, the estimated cost of reaching the goal from $n$ is no greater than the step cost to $n'$ plus the estimated cost from $n'$:
$$h(n) \leq c(n, a, n') + h(n')$$
and the heuristic at the goal state is zero: $h(\text{Goal}) = 0$.

```
         c(n, a, n')
   (n) -------------> (n')
    \                  /
     \                /
 h(n) \              / h(n')
       \            /
        v          v
          (Goal)
```

* **Mathematical Proof of Monotonicity Consequence:** If $h(n)$ is consistent, the sequence of $f$-values along any path is non-decreasing:
  $$f(n') = g(n') + h(n') = g(n) + c(n, a, n') + h(n') \geq g(n) + h(n) = f(n)$$
* **System-Level Benefit:** When A\* runs with a *consistent* heuristic on a graph, once a node is expanded (placed in the closed set), the path to it is guaranteed to be optimal. This prevents the costly overhead of reopening closed nodes.

#### 3. Alpha-Beta Mathematical Reduction
Minimax evaluates $O(b^d)$ states. Alpha-Beta pruning maintains two values:
* $\alpha$: The highest value (best choice) found so far along the path for the **MAX** player.
* $\beta$: The lowest value (best choice) found so far along the path for the **MIN** player.

Pruning occurs when:
$$\beta \leq \alpha$$

```
               MAX [ α = 5, β = +∞ ]
                    /         \
                 /               \
   MIN [ α = -∞, β = 5 ]      MIN [ α = 5, β = 3 ]  <-- Prune branches here!
       /         \                /         \
     (5)         (8)            (3)      (Not Evaluated)
```

* **Optimal Move Ordering Theorem:** If successor nodes are ordered such that the best moves are evaluated first, the branching factor effectively drops from $b$ to $b' = \sqrt{b}$. The search depth doubles for the same computational budget:
  $$\text{Complexity}_{\text{optimal}} = O(b^{d/2})$$

## 2. ⚙️ Under the Hood (Internal Mechanics & Architecture)

### Deep-Dive: A* Implementation Details

To write production-grade pathfinding engine code, you must optimize memory layouts and access patterns:

* **The Priority Queue (PQ) Bottleneck:** The standard heap (`std::priority_queue` or `heapq`) is a binary heap with $O(\log N)$ push/pop. For extremely large spatial graphs, pairing heaps or Fibonacci heaps offer $O(1)$ amortized time for `decrease-key` operations.
* **Tie-Breaking Strategies:** When $f(n_1) = f(n_2)$, the search can wander in flat "plateaus". Correct this by breaking ties in favor of nodes with a **higher $g$ value** (or lower $h$ value). This pushes the search aggressively toward the goal instead of expanding equivalent states near the start:
  ```python
  # Tie-breaking in f-score with secondary evaluation on h-score
  item = (f_score, h_score, node_id) 
  ```
* **Closed-Set Optimizations:** Avoid hashing overhead (`std::unordered_set` or Python `set`) in tight loops. For grid-based terrains, use a flat 1D boolean array (`bitset` or flat contiguous array) indexed by a contiguous spatial ID: $\text{index} = y \times \text{width} + x$. This improves CPU cache locality and eliminates hash collision overhead.
* **State Compression:** Do not store full state structures in the priority queue. Pack your states. For example, a 3D grid coordinate $(x, y, z)$ where $x, y \leq 1024$ and $z \leq 64$ can be bit-packed into a single 32-bit integer:
  $$\text{State} = (x \ll 16) \mid (y \ll 6) \mid z$$

### Deep-Dive: Minimax / Alpha-Beta Optimizations

#### 1. Transposition Tables & Zobrist Hashing
In game-tree searches, identical board states can be reached via different paths (transpositions). Re-evaluating these is highly inefficient.
* **Zobrist Hashing:** We initialize a table of random 64-bit integers for every possible piece-square combination:
  $$\text{Table}[\text{Piece}][\text{Square}]$$
  The hash of a board state is computed incrementally using bitwise XOR ($\oplus$). When a piece moves from $S_1$ to $S_2$:
  $$\text{Hash}_{\text{new}} = \text{Hash}_{\text{old}} \oplus \text{Table}[\text{Piece}][S_1] \oplus \text{Table}[\text{Piece}][S_2]$$
  This operation is $O(1)$ and avoids complete board serialization.
* **Table Entry Layout:**
  ```cpp
  struct TranspositionEntry {
      uint64_t zobrist_key;
      int depth;
      int value;
      enum flag { EXACT, LOWERBOUND, UPPERBOUND } type;
      Move best_move;
  };
  ```

#### 2. Move Ordering (The Lifeblood of Alpha-Beta)
Without good move ordering, Alpha-Beta degrades to raw Minimax. Sort moves using this heuristic hierarchy:
1. **PV Move:** The move from the Principal Variation of the previous search iteration.
2. **Captures / Tactical Moves:** Evaluated using MVV-LVA (Most Valuable Victim - Least Valuable Attacker).
3. **Killer Moves:** Non-capture moves that caused beta-cutoffs in other branches at the same depth.
4. **History Heuristic:** A history table tracking which moves historically caused cutoffs across the entire search tree.

### Industrial Scale Architecture

When routing logistics vehicles or scaling turn-based game servers to millions of concurrent requests, single-threaded algorithms fail. 

```
[API Layer / Client] 
        |
        v
 [In-Memory Cache (Redis/Zobrist State)]
        |
        +---> [Fast Path: Contraction Hierarchies / Highway Networks (O(1) ms)]
        |
        +---> [Dynamic Path: Hierarchical Pathfinding (HPA*) on Grid Tiles]
        |
        +---> [Fallback: Distributed Search / Multi-Threaded IDA*]
```

* **Contraction Hierarchies (CH):** Precomputes "shortcuts" by ordering nodes in a static graph by "importance" and bypassing them. A continental routing query drops from millions of operations in standard A\* to a few dozen in CH, executing in $< 1\text{ ms}$.
* **Hierarchical Pathfinding (HPA\*):** Subdivides large maps into local sectors (e.g., $16 \times 16$ grids). It precomputes gateways between sectors, turning a global pathfinding problem into a high-level search across gateways, followed by a localized low-level search inside the start/end sectors.
* **Iterative Deepening Search & Time Budgeting:** Production game engines run searches under strict time budgets (e.g., $12\text{ ms}$ per turn). They search using Iterative Deepening (Depth 1, then 2, then 3...). When the timer fires, the engine halts execution and returns the best-evaluated move from the deepest completed iteration.

### Production-Grade Code: Alpha-Beta Engine with Transposition Table

Below is a highly optimized, production-grade Python implementation of an Alpha-Beta search engine featuring Zobrist Hashing, a Transposition Table, and Move Ordering.

```python
from typing import Dict, List, Tuple, Optional
import random
import time

# Types for game state representation
Position = int # 0 to 63 (chess-like)
Player = int   # 1 for MAX, -1 for MIN

# Zobrist Hash Initialization
random.seed(42)
ZOBRIST_TABLE: List[List[int]] = [[random.getrandbits(64) for _ in range(64)] for _ in range(2)]

def get_zobrist_hash(board: List[int]) -> int:
    h = 0
    for sq, piece in enumerate(board):
        if piece != 0:
            player_idx = 0 if piece == 1 else 1
            h ^= ZOBRIST_TABLE[player_idx][sq]
    return h


class GameState:
    def __init__(self, board: List[int], turn: Player):
        self.board = board  # 64 elements: 1 for MAX, -1 for MIN, 0 for empty
        self.turn = turn
        self.hash = get_zobrist_hash(self.board)

    def get_legal_moves(self) -> List[Tuple[Position, Position]]:
        """
        Returns moves represented as (start_pos, end_pos).
        """
        moves = []
        for src, piece in enumerate(self.board):
            if piece == self.turn:
                # Mock move generation: adjacent steps for simplicity
                for dst in [src - 8, src + 8, src - 1, src + 1]:
                    if 0 <= dst < 64:
                        moves.append((src, dst))
        return moves

    def make_move(self, move: Tuple[Position, Position]) -> 'GameState':
        src, dst = move
        new_board = list(self.board)
        piece = new_board[src]
        new_board[src] = 0
        new_board[dst] = piece
        return GameState(new_board, -self.turn)

    def evaluate(self) -> int:
        """
        Static evaluation function. Positive favors MAX, negative favors MIN.
        """
        # Feature weight evaluation: raw material + position values
        score = sum(self.board) * 10
        # Positional bias: encourage controlling the center squares (index 27, 28, 35, 36)
        center_squares = {27, 28, 35, 36}
        for sq in center_squares:
            score += self.board[sq] * 2
        return score


class TranspositionTable:
    # Node types
    EXACT = 0
    LOWERBOUND = 1
    UPPERBOUND = 2

    def __init__(self):
        self.table: Dict[int, Tuple[int, int, int, Optional[Tuple[Position, Position]]]] = {}

    def lookup(self, key: int) -> Optional[Tuple[int, int, int, Optional[Tuple[Position, Position]]]]:
        # Returns (depth, value, flag, best_move)
        return self.table.get(key)

    def store(self, key: int, depth: int, value: int, flag: int, move: Optional[Tuple[Position, Position]]):
        # Always replace strategy for simplicity; can be upgraded to depth-preferred replacement
        self.table[key] = (depth, value, flag, move)


class SearchEngine:
    def __init__(self, transposition_table: TranspositionTable):
        self.tt = transposition_table

    def _order_moves(self, state: GameState, moves: List[Tuple[Position, Position]], pv_move: Optional[Tuple[Position, Position]]) -> List[Tuple[Position, Position]]:
        """
        Orders moves to maximize alpha-beta pruning efficiency.
        """
        def move_score(move: Tuple[Position, Position]) -> int:
            if move == pv_move:
                return 1000000  # PV move gets highest priority
            src, dst = move
            # MVV-LVA logic: Capturing an opponent piece has high priority
            target_piece = state.board[dst]
            if target_piece != 0 and target_piece != state.turn:
                return 10000 + abs(target_piece) * 100 - abs(state.board[src])
            return 0

        return sorted(moves, key=move_score, reverse=True)

    def search(self, state: GameState, depth: int, alpha: int, beta: int) -> int:
        alpha_orig = alpha
        
        # Transposition Table Lookup
        tt_entry = self.tt.lookup(state.hash)
        pv_move = None
        if tt_entry is not None:
            tt_depth, tt_val, tt_flag, tt_move = tt_entry
            if tt_depth >= depth:
                if tt_flag == TranspositionTable.EXACT:
                    return tt_val
                elif tt_flag == TranspositionTable.LOWERBOUND:
                    alpha = max(alpha, tt_val)
                elif tt_flag == TranspositionTable.UPPERBOUND:
                    beta = min(beta, tt_val)
                
                if alpha >= beta:
                    return tt_val
            pv_move = tt_move

        # Base case
        if depth == 0:
            return state.evaluate()

        moves = state.get_legal_moves()
        if not moves:
            return state.evaluate()

        # Move Ordering
        ordered_moves = self._order_moves(state, moves, pv_move)
        
        best_move: Optional[Tuple[Position, Position]] = None
        
        if state.turn == 1: # MAX Player
            value = -float('inf')
            for move in ordered_moves:
                next_state = state.make_move(move)
                val = self.search(next_state, depth - 1, alpha, beta)
                if val > value:
                    value = val
                    best_move = move
                alpha = max(alpha, value)
                if alpha >= beta:
                    break  # Beta cutoff
        else: # MIN Player
            value = float('inf')
            for move in ordered_moves:
                next_state = state.make_move(move)
                val = self.search(next_state, depth - 1, alpha, beta)
                if val < value:
                    value = val
                    best_move = move
                beta = min(beta, value)
                if alpha >= beta:
                    break  # Alpha cutoff

        # Store search results in the Transposition Table
        if value <= alpha_orig:
            flag = TranspositionTable.UPPERBOUND
        elif value >= beta:
            flag = TranspositionTable.LOWERBOUND
        else:
            flag = TranspositionTable.EXACT
            
        self.tt.store(state.hash, depth, int(value), flag, best_move)
        return int(value)
```

## 3. ⚠️ The Interview Warzone (Scenarios & Probing Questions)

### Scenario 1: Real-Time Dynamic Fleet Routing for 50k Couriers

#### The Hard Question
> "You are building the real-time routing engine for a global food delivery service with 50,000 active couriers. Every 10 seconds, you receive real-time traffic updates that alter the weights of thousands of edges on your city road graph. How do you construct a low-latency system to dynamically update and calculate paths without burning out our cloud budget?"

```
                       [Spatial Database]
                               |
                   +-----------+-----------+
                   |                       |
       [Live Traffic Engine]               | (Updates)
                   |                       v
                   |          [Dynamic Graph Structure]
                   |                       |
                   |                       v
                   +-------> [Fast Customizer Phase]
                                           |
                                           v
                             [Contraction Hierarchies]
                                           |
                                           v
                             [Parallel Query Thread Pool]
```

#### The Candidate's Trap
A candidate might suggest running traditional A\* or Dijkstra's from scratch for each courier whenever a weight updates. On a real-world city map with $10^6$ nodes, running 50,000 queries every 10 seconds will instantly overload the CPU. 

If they suggest static Contraction Hierarchies, point out that CH requires a costly precomputation step (taking hours) when edge weights change, making it unsuitable for live, dynamic traffic updates.

#### The Probing Pattern
* **Interviewer:** *"If edge weights change dynamically, your Contraction Hierarchy is invalidated. How do you handle that?"*
* **Candidate:** *"I'd recalculate the hierarchy locally."*
* **Interviewer:** *"But how do you define 'locally'? If a central highway gets blocked, it affects routes 10 miles away. What is the algorithmic guarantee?"*

#### The Perfect Response
To solve this, implement **Customizable Contraction Hierarchies (CCH)** combined with **Lifelong Planning A\* (LPA\*)** or **D\* Lite**. 

For continental-scale routing with dynamic traffic, separate the search layout using a **Customizable Contraction Hierarchy (CCH)**:
1. **Metric-Independent Ordering (Phase 1):** Compute the node contraction order based solely on the graph's topology. This is done once (e.g., once a week) and does not change when traffic updates occur.
2. **Customization Phase (Phase 2):** When a batch of traffic updates arrives, run a fast bottom-up weight propagation pass using the static ordering. This takes less than $50\text{ ms}$ for an entire metropolitan area.
3. **Double-Ended Query Phase (Phase 3):** Run a bidirectional Dijkstra search over the customized graph. This resolves queries in $< 1\text{ ms}$.

To handle localized routing updates for active couriers, use **D\* Lite**:
* D\* Lite searches backward from the goal to the courier's start position.
* When edge weights change while the courier is en route, D\* Lite updates only the affected search trees incrementally instead of recalculating the entire path. It re-evaluates the priority queue key values of affected nodes dynamically, ensuring the courier's path remains optimal with minimal computational cost.

### Scenario 2: High-Dimensional Game AI with Massive Branching Factor

#### The Hard Question
> "You are designing the AI engine for a turn-based multiplayer tactical board game. On any turn, a player can move multiple units, yielding a branching factor of $b \approx 20,000$. You have a strict response latency SLA of 100ms. Classic Alpha-Beta search will timeout at depth 2. How do you design this engine?"

```
               [Raw State: b = 20,000]
                          |
                          v
         [Heuristic Filtering / Policy Net]
                          | (Keep Top k=50 Moves)
                          v
             [Pruned State: b_pruned = 50]
                          |
           +--------------+--------------+
           |                             |
    [Search Strategy]             [Evaluation Strategy]
           |                             |
    [Iterative Deepening          [Static Evaluation Model]
     with Alpha-Beta]                    |
                                  [Neural Value Net / Rollout]
```

#### The Candidate's Trap
The candidate might suggest using basic Alpha-Beta pruning or standard Monte Carlo Tree Search (MCTS). However, with $b = 20,000$, standard MCTS will fail to collect meaningful statistics on most branches within 100ms, and Alpha-Beta will suffer from extreme shallow-depth cutoffs without any tactical depth.

#### The Probing Pattern
* **Interviewer:** *"If you prune branches to reduce $b$, how do you guarantee you didn't prune the single winning move?"*
* **Candidate:** *"I'll use a neural network to rank and select the best moves."*
* **Interviewer:** *"Running a deep neural network forward pass 20,000 times will take way longer than 100ms. How do you resolve this?"*

#### The Perfect Response
To design this system within the strict SLA, implement a hybrid search architecture:

1. **Two-Tiered Move Selection:**
   * **Stage 1 (Rapid Filtering):** Use a lightweight, hardware-accelerated linear evaluation or a small convolutional policy network to generate a score for all 20,000 moves. To stay within the 100ms budget, run this evaluation as a parallelized batch job on a local GPU.
   * **Stage 2 (Pruning):** Select only the top $k = 50$ moves for the search tree, reducing the branching factor from 20,000 to a manageable 50.

2. **Principal Variation Search (PVS) with Iterative Deepening:**
   * Run a **Principal Variation Search (PVS)**, which assumes the first ordered move is the best. It searches the remaining moves with a null-window ($\beta = \alpha + 1$) to prove they are worse. If a null-window search fails high, only then do you run a full-width search on that branch. This optimizes pruning efficiency.
   * Wrap the search in an **Iterative Deepening loop** with an elapsed-time check. If the timer hits 85ms, halt the search and return the best move found from the last completed depth iteration.

3. **Heuristic Pruning Techniques:**
   * **Late Move Reductions (LMR):** If a move is sorted deep in the list (low ranking) and searched beyond a certain depth, search it with a reduced depth. If it fails high, re-search it at full depth.
   * **Null Move Pruning:** Allow the engine to "pass" its turn (make a null move). If the resulting evaluation still triggers a beta-cutoff, the position is strong enough that we can prune the entire branch early.

### Scenario 3: Multi-Agent Warehouse Grid Pathfinding

#### The Hard Question
> "In an automated fulfillment center, 1,000 robots move on a shared grid. If we run independent A* searches for each robot, they will collide. If we run a unified A* search in a joint state-space, the state space size scales exponentially as $O(W^{1000})$ where $W$ is the grid size, which is computationally intractable. Design a scalable, collision-free routing algorithm."

```
                         [Global Path Coordinator]
                                     |
                         [Low-Level Search Engine]
                        (Individual A* on Spatiotemporal)
                                     |
                                     v
                       [Conflict Tree Generation]
                                     |
                                     v
                        [High-Level Search Engine]
                     (Resolve Spatiotemporal Conflicts)
                                     |
              +----------------------+----------------------+
              |                                             |
     [Conflict Detected]                             [No Conflict]
              |                                             |
    [Add Spatiotemporal Constraint]                         v
    [Re-route Affected Agent]                       [Dispatch Paths]
```

#### The Candidate's Trap
Candidates often suggest routing robots using token passing or dynamic obstacle avoidance (treating other robots as moving blockages). While simple, this approach causes frequent **deadlocks** (e.g., two robots facing each other in a narrow aisle, neither able to move) and yields highly suboptimal paths.

#### The Probing Pattern
* **Interviewer:** *"If you treat other robots as static obstacles during pathfinding, how do you handle head-on encounters in narrow corridors?"*
* **Candidate:** *"I'll make one robot pull over to a side slot."*
* **Interviewer:** *"How does that robot know a slot is empty and won't block a third robot? How do you coordinate this without creating a circular lock?"*

#### The Perfect Response
To resolve this efficiently, use **Conflict-Based Search (CBS)** or **Windowed Hierarchical Cooperative A\* (WHCA\*)**.

#### Conflict-Based Search (CBS) Architecture
CBS is a two-level search algorithm that decouples agent coordination:

1. **The Low-Level Search:**
   * Run a spatiotemporal A\* search (a 3D space-time grid: $X, Y, \text{Time}$) individually for each agent.
   * The search treats other robots' reserved positions at specific times as invalid states. The cost function is $f(n) = g(n) + h(n)$, where $g(n)$ is the path length and time elapsed.

2. **The High-Level Search:**
   * CBS maintains a **Constraint Tree**. Each node in this tree represents a set of spatiotemporal constraints (e.g., *"Agent $A$ cannot occupy cell $(X_1, Y_1)$ at time $T = 5$"*).
   * It starts with an empty constraint tree. If the low-level paths of two agents conflict (e.g., Agent $A$ and Agent $B$ both try to occupy $(X_1, Y_1)$ at $T=5$):
     * CBS branches the constraint tree into two child nodes:
       1. **Branch 1:** Constraint added: Agent $A$ cannot occupy $(X_1, Y_1)$ at $T=5$.
       2. **Branch 2:** Constraint added: Agent $B$ cannot occupy $(X_1, Y_1)$ at $T=5$.
     * It then updates the low-level paths for the restricted agent in each branch and continues searching the Constraint Tree.

This approach finds the mathematically optimal, collision-free solution for all 1,000 agents. To scale it to real-time operations, implement a **windowed approach (WHCA\*)**: resolve conflicts completely within a sliding time window (e.g., the next 16 timesteps) and recompute paths dynamically as the window advances.
