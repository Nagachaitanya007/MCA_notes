# Engineering Study Note: Data Preprocessing (Outliers, Missing Values, and Skewed Data)

This study note outlines the mathematical foundations, system trade-offs, and production engineering practices required to handle outliers, missing values, and skewed data at scale.

---

## 1. 🧱 The Core Concept

In production machine learning systems, data preprocessing is not a cosmetic step; it directly dictates the optimization landscape of your loss function and the numerical stability of your gradients.

### 1.1 Outliers: Detection and Robustness
An outlier is an observation that deviates significantly from other observations, generated by a different mechanism or extreme stochastic variation.

#### Statistical Outlier Detection Methods
*   **Z-score**: Measures the distance of a data point from the mean in units of standard deviation.
    $$z = \frac{x - \mu}{\sigma}$$
    *   *Assumption*: The underlying feature follows a Gaussian distribution.
    *   *Breakdown Point*: Very low. Because the mean ($\mu$) and standard deviation ($\sigma$) are highly sensitive to extreme outliers, a small cluster of massive outliers will inflate $\sigma$ and pull $\mu$ toward them, masking their own presence (the masking effect).
*   **Interquartile Range (IQR)**: A non-parametric method.
    $$IQR = Q_3 - Q_1$$
    $$\text{Bounds} = [Q_1 - c \times IQR, \,\, Q_3 + c \times IQR] \quad (\text{typically } c = 1.5 \text{ or } 3.0)$$
    *   *Assumption*: Distribution-agnostic, though it assumes symmetry. 
    *   *Breakdown Point*: High. It handles up to 25% arbitrary outliers before the bounds collapse.
*   **Isolation Forest**: An unsupervised tree-based algorithm that isolates anomalies instead of profiling normal data points.
    *   *Mechanism*: It recursively partitions features using random splits. Because anomalies require fewer splits to isolate, they appear closer to the root of the trees.
    *   *Anomalous Score Formulation*:
        $$s(x, n) = 2^{-\frac{\mathbb{E}(h(x))}{c(n)}}$$
        Where $h(x)$ is the path length of sample $x$, $\mathbb{E}(h(x))$ is the expected path length across a forest of trees, and $c(n)$ is the average path length of an unsuccessful search in a Binary Search Tree (BST) built with $n$ nodes:
        $$c(n) = 2 \ln(n - 1) + 0.5772156649 \text{ (Euler's constant)} - \frac{2(n - 1)}{n}$$
        *   If $s \to 1$: Highly likely to be an anomaly.
        *   If $s \ll 0.5$: Structurally normal observation.

### 1.2 Missing Values: Taxonomy and Imputation Mechanics
Data missingness is categorized into three mechanisms (Rubin’s Causal Model):

```
                       ┌─────────────────────────┐
                       │   Missing Data Types    │
                       └────────────┬────────────┘
                                    │
         ┌──────────────────────────┼──────────────────────────┐
         ▼                          ▼                          ▼
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│       MCAR       │       │       MAR        │       │       MNAR       │
│ Missing Completely│       │Missing at Random │       │ Missing Not at   │
│    at Random     │       │                  │       │     Random       │
└──────────────────┘       └──────────────────┘       └──────────────────┘
```

1.  **Missing Completely at Random (MCAR)**: The probability of missingness is entirely independent of both observed and unobserved data.
    $$P(M \mid Y_{obs}, Y_{mis}) = P(M)$$
    *Implication*: Dropping rows (listwise deletion) reduces statistical power but introduces zero bias.
2.  **Missing at Random (MAR)**: The probability of missingness depends systematically on observed variables, but not on the missing value itself.
    $$P(M \mid Y_{obs}, Y_{mis}) = P(M \mid Y_{obs})$$
    *Example*: Younger users are less likely to report their income, but this dependency is fully explained by their age and education level.
3.  **Missing Not at Random (MNAR)**: The probability of missingness depends directly on the unobserved missing value.
    $$P(M \mid Y_{obs}, Y_{mis}) \neq P(M \mid Y_{obs})$$
    *Example*: High-income individuals intentionally hide their salary; extremely sick patients drop out of a clinical trial due to health deterioration. Imputing this without modeling the missingness mechanism introduces severe bias.

#### Imputation Strategies
*   **Mean/Median Imputation**: Replaces missing values with central tendencies.
    *   *Trade-off*: Artificially shrinks feature variance ($\sigma^2 \to 0$) and destroys covariance structures between features, leading to underestimated standard errors and deflated confidence intervals in downstream models.
*   **MICE (Multivariate Imputation by Chained Equations)**: A robust, iterative algorithm that models each variable with missing values conditional on others.
    *   *Step 1*: Impute all missing values using a fast placeholder (e.g., median).
    *   *Step 2*: For feature $X_j$, discard its imputed values, turning it back to missing.
    *   *Step 3*: Train a regressor/classifier $X_j \sim X_{-j}$ (where $X_{-j}$ represents all other features) on observed rows, and predict the missing values of $X_j$.
    *   *Step 4*: Repeat for all features iteratively until convergence (typically 10-20 cycles).
*   **KNN Imputer**: Imputes missing values by calculating the distance-weighted average of the $k$-nearest neighbors in the multi-dimensional feature space.
    *   *Distance Metric (nan-Euclidean)*:
        $$d(x, y) = \sqrt{\frac{d}{d_{present}} \sum_{i \in \text{present}} (x_i - y_i)^2}$$
        Where $d$ is the total number of features, and $d_{present}$ is the number of features present in both vectors.

### 1.3 Skewed Data: Geometric and Statistical Distortions
Skewness measures the asymmetry of a probability distribution around its mean.

$$\gamma_1 = \mathbb{E}\left[\left(\frac{X - \mu}{\sigma}\right)^3\right]$$

*   **Positive (Right) Skew ($\gamma_1 > 0$)**: Mean $>$ Median $>$ Mode. Long tail extends to the right. Common in log-normal distributions (e.g., income, click-through rates, page latency).
*   **Negative (Left) Skew ($\gamma_1 < 0$)**: Mode $>$ Median $>$ Mean. Long tail extends to the left. Common in age of mortality, asset depreciation.

```
       Positive (Right) Skew                     Negative (Left) Skew
       
             ▲                                         ▲
            ╱│╲                                       ╱│╲
           ╱ │ ╲                                     ╱ │ ╲
          ╱  │  ╲                                   ╱  │  ╲
         ╱   │   ╲                                 ╱   │   ╲
        ╱    │    ╲───┐                           ┌───╱    │    ╲
       ╱     │        └───                       ───┘        │     ╲
      ───────┴─────────────►                     ────────────┴──────►
         Mode Median Mean                           Mean Median Mode
```

#### Why Skewness Degrades Model Performance
1.  **Gradient Instability**: In linear and neural network architectures, high-magnitude values in the skewed tail dominate the loss calculation. The gradients corresponding to these extreme values overwhelm those of the typical values, causing the optimizer to oscillate or diverge.
2.  **Violations of Homoscedasticity**: Linear regression models assume constant variance of residuals ($\epsilon \sim \mathcal{N}(0, \sigma^2)$). Heavily skewed target variables yield residuals whose variance scales with the prediction magnitude, violating this assumption and invalidating hypothesis tests (t-tests, F-tests).
3.  **Suboptimal Feature Splits**: For tree-based models, extreme skewness forces splits to occur near the boundaries of the feature range, leading to highly unbalanced trees with poor generalization performance on typical samples.

## 2. ⚙️ Under the Hood

### 2.1 Mathematical Transformations: Deep Dive
When transforming skewed or outlier-heavy data, we apply mathematical mappings to change the spacing of data points.

#### Log Transform
$$y = \ln(x + c)$$
Where $c$ is a small translation constant (typically $1$ to prevent $\ln(0)$). 
*   *Mechanics*: The derivative $\frac{d}{dx}\ln(x) = \frac{1}{x}$ decreases as $x$ increases. This means log transformations heavily compress the upper scale (where outliers reside) while expanding the lower scale, stabilizing variance across different scales.

#### Box-Cox Transformation
A parametric power transformation that searches for a parameter $\lambda$ that transforms the target distribution to as close to normal as possible using Maximum Likelihood Estimation (MLE).

$$y^{(\lambda)} = \begin{cases} \frac{x^\lambda - 1}{\lambda} & \text{if } \lambda \neq 0 \\ \ln(x) & \text{if } \lambda = 0 \end{cases}$$

The objective is to maximize the log-likelihood function $L(\lambda)$ of the transformed data under a normal assumption:

$$L(\lambda) = -\frac{N}{2} \ln(\hat{\sigma}^2(\lambda)) + (\lambda - 1)\sum_{i=1}^N \ln(x_i)$$

Where $\hat{\sigma}^2(\lambda)$ is the sample variance of the transformed data $y^{(\lambda)}$.
*   *Limitation*: Strict constraint $x > 0$.

#### Yeo-Johnson Transformation
Modifies Box-Cox to allow zero and negative values by employing a piecewise power transformation:

$$\psi(\lambda, x) = \begin{cases} 
\frac{(x + 1)^\lambda - 1}{\lambda} & \text{if } \lambda \neq 0, x \ge 0 \\ 
\ln(x + 1) & \text{if } \lambda = 0, x \ge 0 \\ 
-\frac{(-x + 1)^{2 - \lambda} - 1}{2 - \lambda} & \text{if } \lambda \neq 2, x < 0 \\ 
-\ln(-x + 1) & \text{if } \lambda = 2, x < 0 
\end{cases}$$

#### RobustScaler
Unlike StandardScaler, which utilizes $\mu$ and $\sigma$, RobustScaler uses parameters that are resilient to the presence of outliers.

$$x_{scaled} = \frac{x - Q_2(x)}{Q_3(x) - Q_1(x)}$$

*   *Why it is robust*: The median ($Q_2$) and IQR have breakdown points of $50\%$ and $25\%$, respectively. Even if $10\%$ of your data points are extreme outliers, $Q_2$, $Q_1$, and $Q_3$ will remain structurally unaffected.

### 2.2 Data Leakage Mechanics in Preprocessing
Data leakage occurs when information from outside the training dataset is used to configure or train the machine learning model. This leads to overly optimistic cross-validation scores and poor generalization performance in production.

```
❌ BAD: PREPROCESSING ON THE ENTIRE DATASET
┌─────────────────────────────────────────────────────────┐
│                    Entire Dataset                       │
│  (Calculates global μ, σ, median, missing indicators)   │
└────────────────────────────┬────────────────────────────┘
                             ▼
              ┌──────────────┴──────────────┐
              ▼                             ▼
       ┌──────────────┐              ┌──────────────┐
       │  Train Set   │              │   Test Set   │
       └──────────────┘              └──────────────┘

───────────────────────────────────────────────────────────

───
✔️ GOOD: PREPROCESSING STRICTLY FIT ON TRAIN ONLY
              ┌─────────────────────────────┐
              │          Train Set          │
              │ (Calculates μ, σ, median)   │
              └──────────────┬──────────────┘
                             │
            ┌────────────────┴────────────────┐
            ▼                                 ▼
    Apply transform                    Apply transform
            │                                 │
            ▼                                 ▼
     ┌──────────────┐                  ┌──────────────┐
     │ Scaled Train │                  │ Scaled Test  │
     └──────────────┘                  └──────────────┘
```

#### How Scaling Leaks Information
If you compute the global mean $\mu_{global}$ and standard deviation $\sigma_{global}$ on the entire dataset *before* performing your train/test split:

$$x_{scaled\_test} = \frac{x_{test} - \mu_{global}}{\sigma_{global}}$$

The scaled test features now contain information about the distribution of features in the test set (via $\mu_{global}$). If the test set has a different mean, the model implicitly learns details of the test partition during training.

#### How Imputation Leaks Information
Using global median imputation before a split is a common point of failure:

$$x_{imputed} = \text{median}(X_{all})$$

This leaks the target-dependent feature distribution of the validation/test set into the training set, artificially lowering the generalization error.

#### Programmatic Mitigation
Always encapsulate transformations in a pipeline abstraction (e.g., Scikit-Learn `Pipeline` or Spark ML `Pipeline`). 

```python
from sklearn.pipeline import Pipeline
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import RobustScaler
from sklearn.ensemble import RandomForestClassifier

# State is strictly isolated. 
# .fit() calculates parameters (median, IQR) ONLY on the active train split.
# .transform() applies those pre-calculated parameters to the test/validation split.
pipeline = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler', RobustScaler()),
    ('classifier', RandomForestClassifier())
])
```

### 2.3 High-Throughput Production Pipelines
At scale, running offline calculations like $O(N \log N)$ sorting for medians is impossible. Real-time inference systems and petabyte-scale batch frameworks require streaming algorithms and stateless preprocessing.

#### Incremental Calculation of State: Welford's Algorithm
To calculate scaling statistics ($\mu$ and $\sigma^2$) in a distributed, online stream without storing the full dataset in memory, we use Welford’s Algorithm for one-pass variance calculation.

For each new data point $x_k$ at step $k$:

$$M_k = M_{k-1} + \frac{x_k - M_{k-1}}{k}$$

$$S_k = S_{k-1} + (x_k - M_{k-1})(x_k - M_k)$$

$$\sigma^2_k = \frac{S_k}{k - 1}$$

This allows a stateful stream processor (e.g., Apache Flink) to maintain running scaling parameters at $O(1)$ space complexity.

```
       Incoming Data Stream (x₁, x₂, x₃, ...)
                   │
                   ▼
      ┌─────────────────────────┐
      │  Welford's Algorithm    │  ◄── Stores state: k, M_k, S_k (O(1) Memory)
      │  Incremental Updates    │
      └────────────┬────────────┘
                   │
                   ▼
      Updated Scaling Parameters (μ_k, σ²_k)
```

#### Distributed Scaler Execution (Spark / MapReduce)
Under a MapReduce framework:
1.  **Map Stage**: Compute local statistics ($\sum x$, $\sum x^2$, count $n$) for each partition.
2.  **Reduce Stage**: Aggregate local statistics to calculate global mean and variance.
3.  **Map Stage (Transformation)**: Broadcast the global statistics back to worker nodes and apply the scaling transformation to each partition.

## 3. ⚠️ The Interview Warzone

### Scenario 1: Transforming Highly Skewed Click Event Log Data
**Interviewer**: *"We are building a model to predict user ad click-through rates. One key feature is 'time spent on page' which exhibits an extreme positive skew (log-normal) with massive outliers due to bots staying on pages for hours. How do you prepare this feature for a gradient-descent-based neural network model?"*

#### The Red Flags to Avoid
*   ❌ *"I'll use StandardScaler to scale the feature."* (Fails because outliers pull $\mu$ and $\sigma$ to extreme ranges, squashing normal user data into a tight range near zero).
*   ❌ *"I'll use Z-score to drop any user with time > 3 standard deviations."* (Fails because the data is highly non-Gaussian; the mean is heavily skewed, leading to high false positives and dropping valuable real user data).
*   ❌ *"I will apply Box-Cox transformation directly."* (Fails because 'time spent on page' can be exactly 0 for immediate bounces, and Box-Cox requires strictly positive inputs $x > 0$).

#### The Perfect Response
To handle this, I would implement a multi-stage preprocessing pipeline designed for extreme skewness and zero-values while ensuring robustness against bot-generated outliers:

1.  **Apply a Yeo-Johnson Transformation**: Because some users spend exactly 0 seconds on a page (bounces), standard Box-Cox fails. Yeo-Johnson supports zero and negative values while optimizing a power parameter $\lambda$ via Maximum Likelihood Estimation to map the log-normal distribution to a Gaussian shape.
2.  **Robust Scaling**: Instead of standard scaling, I will scale the transformed features using `RobustScaler` (using Median and IQR):
    $$x_{robust} = \frac{x_{YJ} - \text{median}(x_{YJ})}{IQR(x_{YJ})}$$
    This ensures that even if extreme bot behaviors ($10,000+$ seconds) bypass upstream filtering, they will not scale down the variance of typical human interactions ($5$ to $120$ seconds).
3.  **Downstream Loss Function Choice**: Since gradient descent is sensitive to extreme values, I'll pair this with a robust objective function like Huber Loss or Smooth L1 Loss in the neural network to downweight the loss contribution of remaining extreme outliers:
    $$L_\delta(a) = \begin{cases} \frac{1}{2} a^2 & \text{for } |a| \le \delta \\ \delta \left(|a| - \frac{1}{2}\delta\right) & \text{otherwise} \end{cases}$$

### Scenario 2: High-Cardinality Categorical Features with Sparse Missing Data (MNAR)
**Interviewer**: *"We have a transactional fraud detection model. The feature 'merchant_id' has extreme cardinality (500k unique IDs) and is missing for 12% of transactions. Analysis shows transactions without 'merchant_id' are highly correlated with fraud (MNAR). How do you handle this missingness?"*

#### The Red Flags to Avoid
*   ❌ *"I'll run a KNN Imputer to fill in the missing merchant IDs."* (Fails because KNN Imputer on 500k cardinality is a computational bottleneck with $O(N^2 \cdot D)$ complexity; it also violates spatial metric logic for high-cardinality nominal categories).
*   ❌ *"I'll impute missing values with the most frequent merchant_id (Mode)."* (Fails because it destroys the signal that the missingness itself is highly predictive of fraud).
*   ❌ *"I'll drop all rows where merchant_id is null."* (Fails because this drops the key fraud signal you want to capture).

#### The Perfect Response
Since the missingness is classified as Missing Not at Random (MNAR) and is highly correlated with target fraud, we must explicitly model the missingness pattern rather than trying to mask it. I would use the following approach:

1.  **Add a Binary Missing Indicator Column**: Create an explicit indicator feature:
    $$I_{missing} = \begin{cases} 1 & \text{if } \text{merchant\_id is null} \\ 0 & \text{otherwise} \end{cases}$$
    This captures the MNAR signal directly for linear models or neural networks.
2.  **Imputation of a Distinct Category Placeholder**: Fill missing values with a designated string category, such as `"UNKNOWN_MERCHANT"`. This keeps the missing instances grouped during encoding.
3.  **Target Encoding with Empirical Bayes Smoothing**: To address the 500k cardinality without one-hot encoding (which would create an extremely sparse, high-dimensional space), I will apply Target Encoding on the `merchant_id` (including `"UNKNOWN_MERCHANT"`). To prevent overfitting on rare merchants, I will apply smoothing:
    $$S_i = \lambda(n_i) \bar{y}_i + (1 - \lambda(n_i)) \bar{y}_{global}$$
    Where:
    *   $n_i$ is the count of occurrences of merchant $i$.
    *   $\bar{y}_i$ is the average fraud rate for merchant $i$.
    *   $\bar{y}_{global}$ is the global fraud rate.
    *   $\lambda(n_i) = \frac{1}{1 + e^{-(n_i - k) / f}}$ is a sigmoid weight function (where $k$ is a min-samples threshold and $f$ is a smoothing factor).
4.  **Tree-Native Splitting**: If using Gradient Boosted Decision Trees (GBDTs) like LightGBM or XGBoost, I would leverage their native missingness handling. These algorithms assign missing values to whichever split direction minimizes training loss, which natively captures the fraud association without explicit imputation.

### Scenario 3: Mitigating Online Latency and Drift in Production Preprocessing
**Interviewer**: *"You've built a model using RobustScaler and Box-Cox transformations. During an online shopping festival, you see a sudden influx of unseen extreme values and a shift in null rates. Your online service's latency SLA is 10ms. How do you design your pipeline to handle this?"*

```
                         ONLINE INFERENCE PIPELINE (SLA: 10ms)
                                                                            
Incoming Request ──► 1. Schema Validation (Pydantic/Protobuf) ──────┐      
                                                                    ▼      
                     2. Stateful Imputation & Feature Clipping  ◄───┴─── Read Pre-computed Median/Bounds
                                                                         (O(1) Lookup from Redis)
                                                                    │      
                                                                    ▼      
                     3. Fast Transform Execution (Yeo-Johnson/Log) ─┘      
                                                                    │      
                                                                    ▼      
                     4. Model Prediction ───────────────────────────► Result
```

#### The Red Flags to Avoid
*   ❌ *"I'll configure the inference API to run Box-Cox optimization ($\lambda$ estimation) on the fly for each incoming batch."* (Fails because estimating $\lambda$ on the fly is computationally expensive and introduces inconsistency; the online scaling state must be identical to the training state).
*   ❌ *"I will compute the median of the incoming batch to impute missing values in real-time."* (Fails because a single real-time inference request often has a batch size of 1. You cannot calculate median on a single sample, and doing so on mini-batches introduces micro-drift).

#### The Perfect Response
To meet strict low-latency SLAs and handle online drift, we must separate the **state estimation phase** (offline/asynchronous) from the **state application phase** (online/stateless).

1.  **Decouple and Freeze Preprocessing Parameters**:
    *   During offline training, estimate all preprocessing parameters: the Box-Cox parameter $\lambda$, the median $Q_2$, the IQR, and default imputation values.
    *   Serialize these parameters as metadata (e.g., in a JSON config or a model registry) and load them into the online scoring service at startup.
    *   At inference time, the transformations must be strictly mathematical $O(1)$ operations with no statistical estimation:
        $$y = \frac{x^\lambda - 1}{\lambda} \quad (\text{using the fixed offline } \lambda)$$
2.  **Implement Defensive Feature Clipping**:
    To prevent extreme unseen outliers from generating invalid outputs, apply feature clipping using the training IQR limits stored in your metadata:
    $$x_{clipped} = \max(\text{Lower Bound}, \min(x, \text{Upper Bound}))$$
    Where $\text{Lower Bound} = Q_1 - 3 \times IQR$ and $\text{Upper Bound} = Q_3 + 3 \times IQR$.
3.  **Low-Latency Imputation Strategy**:
    If a feature is missing in an incoming online request, use a pre-calculated default imputation value (like the training median) retrieved from a fast in-memory store (e.g., Redis) in $O(1)$ time.
4.  **Continuous Monitoring and Drift Alerting**:
    Set up asynchronous logging to monitor feature drift in real-time. Calculate the **Population Stability Index (PSI)** on windowed chunks of incoming inference features versus the training baseline:
    $$PSI = \sum \left( (P_{actual} - P_{expected}) \times \ln\left(\frac{P_{actual}}{P_{expected}}\right) \right)$$
    If $PSI > 0.25$ (indicating significant distribution shift or a change in missing data rates), trigger an automated workflow to retrain the model and update the preprocessing parameters.
